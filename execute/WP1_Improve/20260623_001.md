# Roadmap: Delivery Verification UI and Line Grouping Enhancements

This document details three specific frontend layout and calculation updates required for the delivery verification modules. Execute these steps sequentially. Maintain existing structural definitions, state hooks, and component boundaries exactly as configured in the project summaries.

---

## Task 1: Unify Single-Batch UI Layouts in DeliveryReceivePage

### Context
In `DeliveryReceivePage.tsx`, transactions with ParentLineNumber 0 can come in two variants: single-batch transactions with an empty or blank BatchNumber, and single-batch transactions with a populated BatchNumber. Currently, they display differently. They need to share the exact same unified single-batch visual style.

### Codebase Changes
1. Locate the item row rendering loop or the `LineItemRow` component in `DeliveryReceivePage.tsx`.
2. Find the conditional rendering checks that inspect `line.parentLineNumber === "0"` (or number equivalent) and `line.batchNumber`.
3. Adjust the UI layout rule: If a row has ParentLineNumber 0 AND the BatchNumber is blank or empty, render it with the exact same tailwind typography, padding, grid columns, and style layout wrappers as a row where ParentLineNumber is 0 and the BatchNumber is NOT blank.
4. Ensure all split-batch interaction panels, child row drawers, or slide actions remain completely untouched and functional for items that undergo split-batch operations.

### Task Status
- [x] Task 1 Completed
- [x] Checked/Updated frontend-summary.md if tracking lines were altered

## Task 2: Refine Items Summary Statistics Calculations

### Context
The top items metric panel summary metrics in `DeliveryReceivePage.tsx` (Total Batches, Accepted, Discrepancy, and Pending) are miscalculating because they are factoring in parent headers from items that have been split out. The statistics need to exclude the parent header of a split item and instead look only at actual distinct physical batches.

### Codebase Changes
1. Locate the `useMemo` calculation logic block that computes summary cards data values inside `DeliveryReceivePage.tsx`.
2. Modify the counting filter array constraints to calculate metrics strictly using this combined definition rule:
   - Count all single-batch transactions where ParentLineNumber is 0 AND the BatchNumber is blank.
   - Count all single-batch transactions where ParentLineNumber is 0 AND the BatchNumber is NOT blank, provided that item line was NOT split into sub-batches.
   - Count all active split child rows (the sub-batch slides themselves, where ParentLineNumber is greater than 0 or marked as child rows), completely ignoring the main parent line row that anchors them.
3. Apply this updated filter logic uniformly across the summary totals for Total Batches, Accepted, Discrepancy, and Pending variables before passing them to the UI data badges.

### Task Status
- [x] Task 2 Completed
- [x] Checked/Updated frontend-summary.md if tracking lines were altered

## Task 3: Grouped Line Items Grid Layout for DeliveryDetailPage

### Context
Currently, `DeliveryDetailPage.tsx` displays line items as a basic, un-grouped table sequence. It needs to adopt the same structured visual grouping hierarchy used inside `DeliveryReceivePage.tsx`.

### Codebase Changes
1. Open `DeliveryDetailPage.tsx` and find the line items rendering section (`_db.DeliveryLines` mapping loop).
2. Implement line item grouping logic inside `DeliveryDetailPage.tsx` modeled after the logic in `DeliveryReceivePage.tsx`: Group sub-batches and child lines cleanly under their respective primary item headers instead of showing them as separate standalone rows.
3. Update the data grid columns to clearly separate parent item attributes from specific batch configurations (such as Batch Number, Delivered Quantity, Returned Quantity, and Rejected Quantity) matching the dashboard style guide.
4. Maintain full pagination compatibility within the detail view file if `totalLinePages` thresholds are crossed.

### Task Status
