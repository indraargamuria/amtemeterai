Target Specification: Invoice Number Storage Alignments and Portable Container Stamping

This document outlines two sequential technical updates required to adapt our e-Meterai stamping execution pipeline to comply with descriptive invoice folder naming and ensure flawless operation across production container environments. Follow the steps sequentially.

Task 1: Align e-Meterai Stamping Endpoint with Invoice Number Hierarchies

Context:
The invoice stamping endpoint currently resolves files based on database primary IDs. This must be refactored to look up, track, and write assets inside object storage using the alphanumeric Invoice Number parameter instead of the database ID, utilizing explicit filename prefixes.

Codebase Changes:

Open your stamping controller containing the route POST /api/invoices/by-sap-number/{invoiceNumber}/stamp.

Update the source file lookup engine. Instead of scanning a folder path driven by an integer ID, change the search path to target the alphanumeric invoice number path structure:
invoices/{invoiceNumber}/printouts/

Locate the file checking logic. Scan the folder contents or resolve the primary document by targeting the descriptive filename prefix format:
INV_{invoiceNumber}_*.pdf
Stream this object down from MinIO as your unsigned base file.

After fetching the stamping serial number and layout coordinates from Peruri PDS, save the resulting QR image asset inside the descriptive invoice folder structure under a dedicated qr subfolder using the following pattern:
invoices/{invoiceNumber}/qr/QRINV_{invoiceNumber}_{GUID}.png

Run the payload through your Peruri signature mapping engine. Configure the output path to save the completed, stamped PDF directly under the stamped subfolder using the following prefix convention:
invoices/{invoiceNumber}/stamped/STPINV_{invoiceNumber}_{GUID}.pdf

Ensure that when writing metadata updates to the database, the StorageKey values match these updated descriptive locations.

Task 2: Refactor Peruri Signing Adapter for Portable Container Execution

Context:
The Peruri adapter requires a local physical workspace layout with subdirectories named UNSIGNED, STAMP, and SIGNED. Mapping this folder directly inside the project root directory causes execution crashes on live container hosts due to immutable image file permissions and access violations. This must be refactored to use dynamic, isolated temporary directories that construct and self-clean natively on any host system.

Codebase Changes:

Locate the file system initialization and variable declarations inside your stamping service layer where the shared physical workspace paths are configured.

Remove any references to hardcoded project root relative paths. Replace them with dynamically constructed system temporary paths using Path.GetTempPath() to ensure cross-platform environment portability:

string stampingSessionId = Guid.NewGuid().ToString("N");
string containerWorkspace = Path.Combine(Path.GetTempPath(), "amtemeterai_stamping", stampingSessionId);
string unsignedDirPath = Path.Combine(containerWorkspace, "UNSIGNED");
string stampDirPath = Path.Combine(containerWorkspace, "STAMP");
string signedDirPath = Path.Combine(containerWorkspace, "SIGNED");

Add robust pre-flight initialization checks to build the temporary workspace folder tree on the host file system before initializing the Peruri execution engine:

Directory.CreateDirectory(unsignedDirPath);
Directory.CreateDirectory(stampDirPath);
Directory.CreateDirectory(signedDirPath);

Stream the base document from MinIO directly into unsignedDirPath, place the retrieved Peruri QR image into stampDirPath, and configure the adapter config parameters to route through containerWorkspace.

After the signed output document has been generated inside signedDirPath and successfully streamed up to your MinIO bucket, wrap the end of the method execution in a comprehensive try-finally block. In the finally loop, completely delete the temporary session workspace folder recursively from the system storage layer to prevent container disk bloating:

finally
{
if (Directory.Exists(containerWorkspace))
{
Directory.Delete(containerWorkspace, true);
}
}
