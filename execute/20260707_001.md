You are an expert Enterprise Systems Engineer and Senior .NET Core Developer. Implement a transactional billing cancellation, locking, and re-billing lifecycle engine between OpexNOW and SAP ERP.

## Task 1: Initialize Context & Read System Summaries
- Read the files `frontend-summary.md` and `backend-summary.md` to understand our architecture, entity models, and existing API routing patterns.
- Acknowledge when done. Do not make code modifications in this step.

## Task 2: Database Schema & Enum Extensions
- **Invoice Status Update (`Models/InvoiceStatus.cs` or equivalent):**
  - Add a new state to the existing Invoice Status Enum: `Voided = 5` (or the next available sequential integer).
- **Delivery Billing Lifecycle Update (`Models/DeliveryHeader.cs`):**
  - Introduce a single new enum tracking column on the model: `public DeliveryBillingStatus BillingStatus { get; set; } = DeliveryBillingStatus.Unbilled;`
  - Define the `DeliveryBillingStatus` Enum with exactly 4 distinct state values:
    - `Unbilled = 1` (Fulfillment confirmed by buyer, ready for initial invoice sync)
    - `Billed = 2` (Invoice successfully generated and synced to SAP)
    - `BillingBlocked = 3` (Invoice voided by SAP; re-billing is strictly prohibited in this lock state)
    - `ReadyToRebill = 4` (SAP has explicitly unlocked the DO, allowing a new invoice simulation run)

## Task 3: Implement the Invoice Void API
- In `Controllers/InvoicesController.cs`, implement a new endpoint: `POST /api/invoices/by-sap-number/{invoiceNumber}/void`
- **Logic requirements (Transactional):**
  - Look up the invoice by its SAP target number.
  - Set `Invoice.Status = InvoiceStatus.Voided`.
  - Traverse the relational link to the associated `DeliveryHeader`.
  - Set that delivery's single tracking state column: `Delivery.BillingStatus = DeliveryBillingStatus.BillingBlocked`.
  - Wrap the entire operation in an explicit DB transaction block to guarantee atomicity.

## Task 4: Implement the Authorization Release API
- In `Controllers/DeliveriesController.cs`, implement a new endpoint: `POST /api/deliveries/by-number/{deliveryNumber}/release-rebill`
- **Logic requirements:**
  - This endpoint will be invoked exclusively by SAP to clear a financial lock.
  - Look up the `DeliveryHeader` via its business identifier (`DeliveryNumber`).
  - Add a guard check: Ensure its current `BillingStatus` is exactly `DeliveryBillingStatus.BillingBlocked`. If it is not, return a `400 Bad Request`.
  - Transition the state: Set `Delivery.BillingStatus = DeliveryBillingStatus.ReadyToRebill`.

## Task 5: Upgrade Core Sync Guardrails
- Open `Controllers/DeliveriesController.cs` and locate the existing Invoice Generation API: `POST /api/deliveries/{deliveryNumber}/invoice`
- **Enforce Business Interlocking Rules:**
  - Inject a strict guard clause at the very beginning of the method processing:
    - If `Delivery.BillingStatus == DeliveryBillingStatus.BillingBlocked`, immediately reject execution and return a `403 Forbidden` response: `"Invoicing is barred while this delivery order remains blocked."`
    - If `Delivery.BillingStatus == DeliveryBillingStatus.Billed`, reject execution to prevent duplicate invoicing attempts.
  - Update successful sync paths: When a delivery with status `Unbilled` or `ReadyToRebill` finishes syncing with SAP successfully, advance its tracking state to `DeliveryBillingStatus.Billed`.
